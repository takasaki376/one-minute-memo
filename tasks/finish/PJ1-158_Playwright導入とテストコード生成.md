# Playwright導入とテストコード生成

**Jiraチケット用テキスト案**

## タイトル
Playwrightの導入とE2Eテストコードの作成

## 概要
手動での動作確認コストを削減し、リグレッションを防ぐために、E2EテストツールのPlaywrightを導入します。
主要なユーザーストーリー（セッション開始〜完了、履歴閲覧、設定変更）をカバーするテストコードを作成し、自動テストの基盤を整えます。

## 目的
- ブラウザ操作を伴うE2Eテストを自動化する
- デプロイ前やPR作成時に主要機能が壊れていないかを機械的に保証できるようにする

## 作業内容

### 1. Playwrightのインストールとセットアップ
- `yarn create playwright` 等での導入
- `playwright.config.ts` の設定（ベースURL、ブラウザ設定、スクリーンショット設定など）
- `package.json` へのスクリプト追加 (`test:e2e`, `test:e2e:ui` 等)

### 2. テスト構成の検討
- Page Object Model (POM) を採用するか、シンプルな構成にするかの決定
- テストデータの扱い（IndexedDBのモック or リセット処理の方法）の確立
  - ※本アプリはIndexedDBを使用しているため、テスト実行ごとのクリーンアップ処理が重要

### 3. テストシナリオの実装
まずは以下のハッピーパスを実装する：
- **セッション実行フロー**:
  - トップ画面 -> セッション開始 -> メモ書き込み（テキスト・手書き） -> 次のテーマへ -> セッション完了 -> 完了画面表示
- **履歴確認フロー**:
  - 履歴一覧画面 -> 履歴詳細画面 -> 保存したメモの内容確認
- **設定変更フロー**:
  - 設定画面 -> テーマ数変更 -> 保存 -> セッション開始時に反映されているか確認

### 4. CI連携（Optional / 後回し可）
- GitHub Actions での実行設定

## 完了条件
- `yarn test:e2e` コマンドで主要シナリオのテストが成功すること
- テスト失敗時にレポート（スクリーンショット/動画）が確認できること
- 既存の主要機能（セッション、履歴、設定）の動作がE2Eテストでカバーされていること

## 見積もり
- 導入・設定: 2h
- テストコード実装: 4h
- 合計: 6h

## 補足：実装メモ

### インストールコマンド
```bash
yarn create playwright
```

### 考慮事項
- **IndexedDBのハンドリング**: Playwrightはブラウザコンテキストが独立しているため基本はクリーンだが、明示的にクリアするヘルパーを用意しておくと安心。
- **モバイルビュー**: スマホ利用がメインのアプリであれば、`projects` 設定で `Mobile Chrome` などのデバイスエミュレーション設定を有効にする。
