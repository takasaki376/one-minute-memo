## Task 6-2: useSettings の単体テストを実装する（詳細）

### Sub-task 6-2-1: settingsRepo をモックする

- getSettings / updateSettings / resetSettings を vi.fn で差し替える
- 成功/失敗パターンを作れるようにする

### Sub-task 6-2-2: 初回ロード成功のテスト

- 初期:
  - isLoading=true
- getSettings resolve 後:
  - isLoading=false
  - settings がセットされる
  - error が null

### Sub-task 6-2-3: 初回ロード失敗のテスト

- getSettings reject
- isLoading=false
- error がセットされる
- settings の扱い（null or default）を仕様通りに確認

### Sub-task 6-2-4: updateSettings 成功のテスト

- updateSettings(patch) 実行で settings が更新される
- error がクリアされる

### Sub-task 6-2-5: updateSettings 失敗のテスト

- updateSettings reject
- error がセットされる
- settings は直前値のまま（フォーム維持方針と整合）

### Sub-task 6-2-6: resetSettings 成功/失敗のテスト（採用時）

- reset 成功で default 反映
- reset 失敗で error 表示

## Task 6-3: SettingPage のコンポーネントテストを実装する（詳細）

MVP 要件: 更新失敗してもフォームは残り、上部に簡易エラー表示が出る。

### Sub-task 6-3-1: useSettings をモックする

- 画面テストは hook をモックして「状態別の描画」を確認する
- パターン:
  - loading
  - 初期ロード error
  - 正常（settings あり）

### Sub-task 6-3-2: loading 表示のテスト

- isLoading=true のとき Loader が表示される
- 入力フォームが表示されないこと（期待仕様なら）

### Sub-task 6-3-3: 初期ロード error 表示のテスト

- error あり（初期ロード失敗）
- エラー文言が表示され、入力フォームが出ないこと

### Sub-task 6-3-4: 正常表示のテスト

- theme_count / time_limit が入力に表示される
- 推定所要時間が表示される
- 「次回セッション開始から適用」の注記が表示される
- 閉じるボタンが `/` にリンクしていること

### Sub-task 6-3-5: onBlur 更新成功のテスト

- theme_count を変更 → blur → updateSettings が呼ばれる
- update 成功時に上部エラーが出ない（または消える）

### Sub-task 6-3-6: onBlur 更新失敗のテスト（MVP 中核）

- updateSettings を reject させる
- blur 後に上部に簡易エラー表示が出る
- フォーム（入力欄）が残っている（操作継続できる）こと

### Sub-task 6-3-7: 初期値に戻すのテスト

- resetSettings または updateSettings が呼ばれる
- 成功時に値が初期値に戻る
- 失敗時に上部エラー表示が出てフォームが残る

## Task 6-4: /session 連携の回帰テストを更新する（詳細）

目的: settings が /session のテーマ数・タイマー秒数・完了条件に影響することを担保する。

### Sub-task 6-4-1: settingsRepo.getSettings をモックできるようにする

- /session が repo 直呼びの場合、vi.mock で差し替え可能にする

### Sub-task 6-4-2: テーマ数が settings に従うことのテスト

- getSettings を `{ theme_count: 3, time_limit: "60" }` にモック
- 3 回進めたら completeSession が呼ばれる
- router.push('/session/complete') が呼ばれる

### Sub-task 6-4-3: タイマー秒数が settings に従うことのテスト

- getSettings を `{ theme_count: 10, time_limit: "30" }` にモック
- useCountdown が initialSeconds=30 で呼ばれること
  - useCountdown をモック/spy して引数を検証

### Sub-task 6-4-4: settings 取得失敗時のフォールバックテスト

- getSettings reject
- /session が落ちずに初期化される
- デフォルト（10 テーマ/60 秒）相当で進行できる（少なくともタイマー初期値が 60）

### Sub-task 6-4-5: “次回適用” の担保テスト（任意）

- 1 回目の /session 初期化後に settingsRepo.getSettings の返り値を変更しても、
  進行中セッションの initialSeconds が変わらない（都度参照しない）ことを確認
- 実装が「開始時に値を固定」していれば、このテストで回帰を防げる
