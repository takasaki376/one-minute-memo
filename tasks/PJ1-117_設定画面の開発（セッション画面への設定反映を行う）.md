## Task 5-1: セッション開始時に Settings を読み込む（詳細）

### Sub-task 5-1-1: Settings 取得方法を決定する

- /session では useSettings を使わず、repo から直接取得する方針にする（推奨）
  - 理由: “開始時点の値を固定” しやすい、UI 状態（loading/error）の取り回しが単純
- 使用する関数:
  - `settingsRepo.getSettings()` もしくは `useSettings` の内部関数

### Sub-task 5-1-2: /session 初期化フローに Settings 取得を追加する

- ページマウント時（useEffect）に以下を実行
  1. `getSettings()`（保存済み設定）
  2. 設定値を state として保持（このセッションで固定）
  3. テーマ抽選 → createSession → タイマー開始
- Settings 読み込み前に処理が進まないように制御する（ガード）

### Sub-task 5-1-3: ローディング状態を整備する

- Settings 読み込み中は /session を「準備中」表示にする
- 既存のローディング表示がある場合は統合する

### Sub-task 5-1-4: Settings 読み込み失敗時のフォールバックを実装する

- getSettings が失敗してもセッション開始できるようにする（MVP 安定性）
  - デフォルト値: theme_count=10 / time_limit=60
- エラーは console.error に出す（必要なら UI に軽い警告を出す）

## Task 5-2: テーマ抽選ロジックに設定（theme_count）を反映する（詳細）

### Sub-task 5-2-1: 既存のランダム抽選関数の入出力を確認する

- 既存: 200 テーマから 10 件選ぶ（固定）
- 変更: 200 テーマから `theme_count` 件を選ぶ

### Sub-task 5-2-2: 抽選関数の引数を拡張する

- 例: `pickRandomThemes(allThemes, count)`
- count は settings.theme_count を使用

### Sub-task 5-2-3: エッジケース対応を追加する

- theme_count が全テーマ数を上回る場合
  - クランプして `min(count, allThemes.length)` にする
- テーマが 0 件の場合（初期投入失敗など）
  - セッション開始をブロックし、エラーメッセージ表示（MVP では最低限の警告）

### Sub-task 5-2-4: セッション作成に渡す themeIds を設定値に基づき生成する

- `const themeIds = picked.map(t => t.id)`
- createSession に渡す配列が `theme_count` 件になっていることを確認

## Task 5-3: タイマー初期値に設定（time_limit）を反映する（詳細）

### Sub-task 5-3-1: time_limit の型を整理する

- time_limit が string の場合:
  - `const seconds = Number(time_limit)`
  - NaN の場合はデフォルト 60 を採用
- time_limit が number の場合はそのまま

### Sub-task 5-3-2: useCountdown への initialSeconds 反映を実装する

- `useCountdown({ initialSeconds: secondsPerTheme, ... })`
- 1 テーマごとに reset する設計の場合、reset に渡す秒数も同値になるよう統一する

### Sub-task 5-3-3: 1 テーマ終了時のタイマー再初期化を確認する

- 次テーマ開始時に
  - reset が走る
  - secondsLeft が設定秒数に戻る
- 設定変更が “進行中セッションに影響しない” ように
  - secondsPerTheme は session init 時の state を参照する（settings を都度読まない）

## Task 5-5: “次回セッション開始から適用” を担保する（詳細）

### Sub-task 5-5-1: /session で settings を “毎テーマ” 読まないことを確認する

- セッション開始時に 1 回読み、以後は固定値 state を参照する
- 進行中に /setting で変更しても、このセッションに影響しない

### Sub-task 5-5-2: セッション開始と設定取得の順序を固定する

- 必ず settings 取得 → themeIds 生成 → createSession の順
- race condition が起きないように async を直列化する

### Sub-task 5-5-3: 手動テスト項目を整理する

- セッション開始 → 途中で settings 変更 → セッションが変わらない
- セッション完了 → 新規開始 → 新設定が反映される
