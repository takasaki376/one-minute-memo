## 1. 目的・コンセプト

- **目的**

  - ゼロ秒思考を簡単に・継続しやすくする。
  - 1 回あたり約 10 分で、10 個前後のテーマについて 1 分ずつ集中して書ける。
  - 過去のメモを見返すことで、自分の思考や感情のパターンを把握・改善する。

- **ターゲット・利用シーン**

  - ビジネスパーソンや学生など、ゼロ秒思考を実践したい人。
  - 朝／仕事前／就寝前など、短時間の思考タイム用。

---

## 2. プラットフォーム・技術前提

- **プラットフォーム**

  - PC ブラウザ向け Web アプリ（将来的にスマホ対応も視野に入れる）。

- **技術スタック（想定）**

  - フロント：Next.js（React）
  - データ保存：

    - v1：ブラウザ内の IndexedDB
    - 将来：Supabase（クラウド DB）＋ログイン認証

---

## 3. コア機能（MVP）

### 3-1. テーマ管理

- 初期テーマ：200 件をアプリ内にプリセット。
- セッション開始時：

  - 有効なテーマの中からランダムで 10 件を選出（重複なし）。

- 必要な項目：

  - テーマ ID
  - テーマ名
  - 有効フラグ（出題する／しない）

※ユーザー独自テーマ追加やカテゴリは、将来拡張側に回して OK。

---

### 3-2. セッション機能（ゼロ秒思考タイム）

- セッション開始操作：

  - 「セッション開始」ボタンなどで開始。
  - その時点で「今回使うテーマ一覧（例：10 個）」が決定される。

- 進行：

  - 1 テーマにつき 1 分のタイマー。
  - カウントダウン表示（残り秒数）。
  - 1 分経過したらそのテーマの入力をロックし、自動で次のテーマに切り替え。
  - 全テーマ分が終了すると「セッション終了」画面へ。

- 一時停止・中断（MVP 方針）：

  - 基本は**一時停止なし**（ゼロ秒思考の緊張感を維持）。
  - ウィンドウを閉じたりリロードした場合の扱いは「セッションは終了として扱う」「途中までの入力は保存される」でよい。

---

### 3-3. 入力画面

1 つのテーマについての入力画面で、下記を表示：

- 現在のテーマ名

- 残り時間（カウントダウン）

- 入力エリア

  - **キーボード入力用テキストエリア**
  - **手書き入力パッド**

    - マウス／ペンタブ／タッチなどで書けるキャンバス
    - 手書き内容は画像データとして保存（IndexedDB に格納）
      ※文字認識までは v1 ではやらない想定でも OK

- 操作のイメージ：

  - できるだけキーボード操作のみでテーマを進められる（Enter で次へなど）※詳細は後で UI 設計。
  - タイマーが 0 になったら：

    - テキスト入力・手書き入力とも編集不可（入力終了状態）。
    - 2 秒程度のインターバル表示（「次のテーマへ」）を挟むか、そのまま自動遷移かを要件で決める。

---

### 3-4. 履歴・見返し機能

- 単位：**セッション**ごとに履歴が作成される。

  - セッション一覧画面：

    - 実施日・開始時刻
    - 使用テーマ数（基本 10）

  - セッション詳細画面：

    - そのセッションで書いた各テーマごとのメモ一覧
      （テーマ名＋テキスト＋手書き画像サムネイル）

- 最低限の検索・フィルタ（MVP）：

  - 日付ごとのフィルタ（カレンダー or 日付リスト）。
  - テーマ名ベースの検索。

## 4. データ構造・保存要件（v1：IndexedDB）

### 4-1. テーマ

- `id`: string / number
- `title`: string
- `isActive`: boolean
- （将来拡張用）`category`, `source`（初期／ユーザー追加）など

### 4-2. セッション

- `id`: string
- `startedAt`: Date
- `endedAt`: Date
- `themeIds`: string[]（そのセッションで使ったテーマ ID の配列）

### 4-3. メモ

- `id`: string
- `sessionId`: string
- `themeId`: string
- `textContent`: string（テキスト入力）
- `handwritingData`:

  - 画像として保存する場合：Blob / DataURL
  - 将来、Stroke データとしてベクタ形式保存する案もあり

- `createdAt`: Date
- `updatedAt`: Date

### 4-4. 保存場所

- v1 ではすべて IndexedDB に保存。

  - オフラインでも利用可能。
  - ブラウザごとのローカルデータとなる。

## 5. 設定・カスタマイズ

MVP 時点では最低限に絞る：

- セッション関連

  - テーマ数：固定 10（将来変更可能に）
  - 1 テーマあたり時間：固定 60 秒（将来変更可能に）

- テーマの ON/OFF

  - 初期テーマの中で「出題対象から外す」チェック機能。

- 表示

  - 文字サイズ（中／大くらい）
  - ダークモードは将来拡張でも OK。

---

## 6. セキュリティ・プライバシー

- v1 時点：

  - データは IndexedDB にのみ保存され、サーバには送信されない。
  - 「このブラウザにのみ保存される」ことを明示。

- 通信：

  - Web アプリとしては HTTPS を前提にデプロイ。

## 7. 将来的な拡張（Supabase 連携・ログイン）

ここは「v2 以降の要件」として整理しておく部分です。

### 7-1. ログイン・ユーザー管理

- 認証

  - Supabase Auth（メール＋パスワード、または OAuth など）を利用。

- ユーザー ID 単位でクラウドにセッション／メモ／テーマ設定を紐づけ。

### 7-2. クラウド同期フロー

- 基本方針：

  - v1 では IndexedDB に保存。
  - 将来、**セッション終了時に非同期で Supabase にアップロード**する機能を追加。

- 想定フロー：

  1. ユーザーがログインしている状態でセッションを実施。
  2. セッション終了時、IndexedDB に保存したデータを基に Supabase へ同期。
  3. ネットワークエラー時は後で再送（再同期キュー）できるように設計。

- 複数デバイス対応：

  - ログインしたユーザーは、どのデバイスからでも過去セッションを閲覧できる。
  - クラウド側のデータ > ローカル側データ のようなマージルールを決めておく。

    - 初期は「クラウド＝マスタ」「ローカルはキャッシュ」くらいのシンプルな方針でも OK。

### 7-3. 手書きデータの保存戦略（拡張）

- Supabase に保存する場合：

  - 手書き画像は Storage（オブジェクトストレージ）にアップロード。
  - DB にはその URL とメタ情報を保存。

- 大量データ化を見越して：

  - １ユーザーあたり容量の目安や削除ポリシー（古いものを削除／エクスポート）。
